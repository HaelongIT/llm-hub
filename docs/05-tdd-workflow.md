# 05 · TDD 워크플로우

> 모든 기능 코드는 테스트가 먼저다. 이 절차를 어기고 구현부터 짜지 않는다.

## 기본 사이클 (Red → Green → Refactor)

1. **Red** — 요구사항(`requirements/REQ-*.md`)의 테스트 시나리오 중 하나를 골라 **실패하는 테스트**를 먼저 작성. 실행해서 **실제로 실패하는지 확인**(이유가 "미구현"이어야지 "오타/설정오류"면 안 됨).
2. **Green** — 그 테스트를 통과시키는 **최소** 구현. 과설계 금지.
3. **Refactor** — 중복 제거·경계(E) 정리. 테스트가 계속 통과하는지 확인.
4. **교훈 기록** — 배운 것을 `docs/LEARNINGS.md`에 남긴다(`docs/06`).
5. **커밋** — 테스트+구현을 함께 커밋(`docs/07`).

한 번에 하나의 시나리오. 여러 개를 몰아서 구현하지 않는다.

## 테스트 계층

| 계층 | 대상 | 도구(예시) | 외부 의존 |
|---|---|---|---|
| **단위** | 순수 로직(청킹, 태그 매핑, 병합, 컨텍스트 조립) | JUnit5 + Mockito | 전부 mock |
| **슬라이스** | 리포지토리·컨트롤러 단위 | Spring test slices, WebTestClient | 최소 |
| **통합** | ES·PG·Keycloak 실제 연동 | **Testcontainers** | 컨테이너 |
| **계약** | 모듈 간 인터페이스(E 경계) | 인터페이스 대체 구현 테스트 | mock/stub |

- 외부 의존(ES·LLM·Keycloak)은 인터페이스로 추상화(MAINT-2)해 단위 테스트에서 대체 가능하게.
- LLM 호출은 단위·슬라이스에서 stub. 실제 게이트웨이 호출은 별도 통합 테스트(선택).
- ES 하이브리드 검색·nori 형태소 같은 "실제 엔진 동작"에 의존하는 것은 **Testcontainers 통합 테스트**로.

## 모듈 구현 순서 (의존성 기반)

각 모듈은 독립 테스트 가능하지만, 통합 흐름 검증을 위한 권장 순서:

1. **IDX** — 색인이 없으면 검색할 게 없다. 데이터 스키마·불변식부터.
   - 파서 → 청킹 → 임베딩(stub) → 저장 → 교체 순서.
2. **AUTH** — 태그 확정 게이트. SEARCH가 이 태그에 의존.
   - RoleTagMapper 단위 → 필터체인 슬라이스 → 태그 적재.
3. **SEARCH** — IDX 데이터 + AUTH 태그를 소비.
   - 하이브리드 쿼리(Testcontainers) → 태그 필터 → sources 생성.
4. **CHAT** — AUTH·SEARCH를 오케스트레이션.
   - 컨텍스트 조립 → 이벤트 SSE → 이력 저장(격리) → 장애 처리.
5. **AUDIT** — CHAT 완료 훅.
   - 독립 기록 → FK 없음 검증 → 삭제 무관성.

## 검증 체크리스트 매핑

`docs/00-overview.md`의 v0 완성 체크리스트 항목 = 통합 테스트로 최종 검증. 각 REQ의 시나리오를 다 통과한 뒤, 체크리스트를 end-to-end 통합 테스트로 확인한다.

## 규칙
- 테스트가 없으면 구현을 병합하지 않는다.
- 실패하는 테스트를 통과시키려 **테스트를 약화시키지 않는다**(단언 삭제·조건 완화 금지).
- 커버리지 수치보다 **불변식·경계·정합성**을 테스트하는 것이 우선.
- 테스트가 통과하지 않는 채로 다음 시나리오로 넘어가지 않는다.

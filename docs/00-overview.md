# 00 · 프로젝트 개요

## 정체성

회사별 설치형 배포를 전제로 한, 인증·권한·감사가 내장된 **도메인 중립 RAG 챗 런타임**. 도메인 로직은 코어에 담지 않고, 확장 지점을 계약(인터페이스)으로 노출한다.

원칙:
- **코어의 순수성** — 특정 도메인 개념(규정·인사·ERP)은 코어에 없다.
- **도메인 중립 ≠ 언어 중립** — 코어는 도메인을 가정하지 않되, 언어는 배포 설정이다(v0=한국어, nori 분석기). 언어 특화 부품은 교체 가능해야 한다.
- **확장을 막지 않는다** — v0에서 "안 만든다"는 결정은 많지만, "나중에 못 만들게 된다"는 결정은 없어야 한다.
- **참조 모델** — Open WebUI를 Spring 생태계로 대체. 방향성은 유사하되 기능 패리티 복제는 아니다.

## v0 동작 흐름 (end-to-end)

1. **문서 유입·색인** — 관리자가 색인 API로 문서 업로드 → Tika/hwplib 추출 → 토큰 청킹 → 고정 임베딩으로 벡터화 → 필수 메타데이터 + 원문과 함께 Elasticsearch 저장. 원본 파일 보관.
2. **인증·태그 확정** — Keycloak 로그인 → Security 필터체인(CORS→JWT→RBAC→RateLimit)의 RBAC 단계가 역할→접근태그를 확정해 요청 컨텍스트에 적재.
3. **질문·검색·생성** — 세션에서 질문 → 최근 N턴 컨텍스트 조립 → 접근태그로 필터된 하이브리드 검색(BM25+벡터) → 단일 고정 LLM으로 근거 기반 답변 생성.
4. **응답 스트리밍** — 이벤트 SSE: `text`(조각) + `sources`(서버 검색 결과) + `error`(장애) + `done`.
5. **감사 기록** — 대화 이력과 별도로 감사 로그 기록.

## v0 완성 검증 체크리스트

이 항목들이 모두 통과하면 v0 완성. 각 항목은 `docs/05-tdd-workflow.md`에서 테스트 시나리오로 구체화된다.

- [x] 관리자가 색인 API로 PDF/txt/hwp 문서를 넣으면 검색 가능한 상태로 색인된다.
- [x] 색인된 각 조각에 필수 메타데이터(문서ID·명·위치·접근태그·색인시각·임베딩모델·차원)가 모두 있다.
- [x] 로그인한 사용자가 질문하면 근거(sources)가 붙은 답이 스트리밍으로 나온다.
- [x] sources에 담긴 근거는 서버의 실제 검색 결과이며, LLM이 지어낸 것이 아니다.
- [x] 권한 없는 사용자에게는 restricted 문서가 검색 결과에 아예 나오지 않는다.
- [x] 태그 확정이 RAG 진입 전(앞단 게이트)에서 이루어진다.
- [x] 후속 질문 시 최근 N턴 대화가 컨텍스트에 반영된다.
- [x] 정확한 용어(고유명사·코드·숫자)로 물어도 해당 문서가 검색된다(BM25+벡터 하이브리드, 한국어 형태소).
- [x] LLM/ES 장애 시 스트림이 `error` 이벤트로 깔끔히 닫히고 사용자가 상황을 안다.
- [x] 사용자가 세션(대화 이력)을 삭제해도 감사 로그는 남는다.
- [x] 같은 문서 키로 재업로드하면 구버전 조각이 사라지고 신버전만 검색된다.
- [x] 부하 시에도 블로킹 DB 접근이 논블로킹 스트리밍 흐름을 막지 않는다.
- [x] 색인 API는 인증된 관리자(ADMIN)만 호출할 수 있다.

# 04 · 비기능 요구사항 (NFR)

> 기능이 "동작하는가"와 별개로 반드시 충족해야 하는 품질 속성. 각 항목은 테스트·리뷰로 검증한다.

## 보안 (SEC)

### SEC-1 · 인증
- 모든 API는 인증 필요(색인 API 포함). 인증되지 않은 요청은 401.
- JWT 검증 실패·만료 시 거부. Refresh는 Keycloak 위임.
- BFF만 외부 노출. LiteLLM Gateway·Elasticsearch·PostgreSQL은 내부망 전용, 외부 미노출.

### SEC-2 · 인가 (권한)
- 접근 태그 확정은 **앞단 게이트(RBAC 단계)에서만**(S4). 하위 계층은 태그 소비만.
- 검색은 사용자 접근 태그로 **반드시** 필터링. 권한 없는 조각은 검색 결과에 포함되지 않는다(응답에서 제거가 아니라 **검색 단계에서 배제**).
- 색인 API는 ADMIN 역할만.
- 권한 검사 누락 가능 경로가 없어야 함(모든 요청이 게이트 통과 구조).

### SEC-3 · 비밀 관리
- 키·토큰·비밀번호·DB 자격증명은 코드·저장소에 **절대** 포함 금지.
- 설정은 환경변수. `.env`는 gitignore, `.env.example`만 커밋(값은 placeholder).
- 로그에 자격증명·토큰·개인정보 원문을 남기지 않는다.

### SEC-4 · 입력 검증
- 색인 업로드: 크기·확장자·MIME 검증. 허용 목록 방식.
- 사용자 입력(질문)은 프롬프트 인젝션을 고려하되, v0는 근거를 서버 검색으로 강제(S6)해 LLM이 지어낸 근거를 차단.
- 경로 조작(path traversal) 방지: 원본 저장 경로는 시스템 생성, 사용자 입력 파일명을 그대로 경로에 쓰지 않는다.

### SEC-5 · 데이터 보호
- 감사 로그 접근은 감사 권한자만(v0는 최소 구현, 접근 통제 자리 확보).
- 대화 이력은 사용자 소유·삭제 가능. 감사 로그는 별도 보존(S5).
- 개인정보 최소 수집: 필요한 필드만 저장.

### SEC-6 · 격리
- 06 워커(로드맵)는 격리 컨테이너·화이트리스트 연산·임의 코드 실행 없음 — v0엔 없으나 도입 시 이 원칙.

## 성능 (PERF)

### PERF-1 · 논블로킹 무결성
- WebFlux 논블로킹 흐름을 블로킹 호출이 막지 않는다(S13). 블로킹 DB·파일 I/O는 격리 스케줄러(boundedElastic 등)에서만.
- 스트리밍 응답은 첫 토큰까지 지연 최소화. 이력·감사 저장은 스트림을 지연시키지 않는다(비동기).

### PERF-2 · 스트리밍
- 응답은 `text` 이벤트로 점진 전달(사용자 체감 지연 감소).
- 스트림 도중 장애 시 즉시 `error` 이벤트로 종료(무한 대기·행 방지).

### PERF-3 · 검색
- 하이브리드 검색은 단일 ES 쿼리로(BM25+벡터를 앱 레벨에서 N회 반복 호출하지 않음).
- top-k는 설정값. 과도한 조각 반환으로 컨텍스트 폭증 방지.

### PERF-4 · 색인
- v0 색인은 동기 처리(소규모 전제). 대용량은 로드맵(비동기 큐).
- 재색인은 원본에서 수행하되, 교체 시 "신버전 완료 후 구버전 삭제"로 검색 공백 최소화(S17).

### PERF-5 · 자원
- 컨텍스트 윈도우는 최근 N턴으로 제한(무한 누적 방지).
- LLM 컨텍스트 길이 초과를 방지하는 상한(설정).

## 신뢰성 / 운영 (REL)

### REL-1 · 장애 처리
- LLM·ES·게이트웨이 장애 시 깨끗한 실패(S8-3): 명확한 에러 + 로깅 + `error` 이벤트. 부분 응답 후 끊김을 사용자가 구분할 수 있어야.
- 재색인 중 장애 시 문서 증발 없음(S17 순서 보장).

### REL-2 · 데이터 정합성
- `docs/03` 불변식을 위반하는 상태가 만들어지지 않는다.
- 저장 실패(이력·감사)는 로깅하되 사용자 응답을 되돌리지 않는다. 단 감사 저장 실패는 경고 로그로 눈에 띄게.

### REL-3 · 관측성
- 요청 단위 추적 ID(trace_id) 발급, 감사·로그에 연결.
- 주요 단계(색인·검색·LLM 호출·저장) 로깅. 실패는 원인 식별 가능하게.

### REL-4 · 설정 외부화
- 임베딩 모델, N턴, top-k, 분석기, RateLimit, 저장 경로 등은 설정값. 코드 하드코딩 금지.

### REL-5 · 배포 재현성
- 로컬 Docker Compose로 전체 스택 기동 가능(개발·검증). 회사별 설치형과 동일 산출물.

## 유지보수 (MAINT)

### MAINT-1 · 모듈 경계
- `docs/02` 경계(E)를 인터페이스로 구현. 교체 가능 지점은 인터페이스 뒤에 숨긴다.
- 모듈 간 순환 의존 금지.

### MAINT-2 · 테스트 가능성
- 외부 의존(ES·LLM·Keycloak)은 인터페이스로 추상화해 테스트에서 대체 가능(mock/testcontainers).
- 각 모듈은 독립적으로 단위 테스트 가능해야 한다(`docs/05`).

### MAINT-3 · 문서 동기화
- 결정을 바꾸면 `docs/02`와 관련 요구사항 문서를 함께 갱신. 코드와 문서 불일치 금지.

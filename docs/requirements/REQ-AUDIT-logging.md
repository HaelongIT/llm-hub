# REQ-AUDIT · 감사 로그

## 책임
채팅 요청·응답·근거를 감사 목적으로 기록한다. 대화 이력과 독립적인 수명주기.

## 관련 결정
S5 · 경계 E5 · NFR SEC-5, REL-2, REL-3

## 인터페이스
- `AuditLogRepository`(PG, 블로킹 → 격리 계층).
- 감사 기록은 CHAT 응답 완료 시점에 비동기 트리거.

## 기능 명세
1. **기록 시점** — 응답 스트림 완료 시(격리 스케줄러). 응답 성공과 독립.
2. **레코드 내용** — `trace_id`, `requester_id`(값 복사), `question`, `answer`, `sources_json`, `created_at`. 기록 범위(전문/최소)는 설정(E5) — v0 기본 전문.
3. **독립 저장** — `audit_log` 테이블, **세션·사용자 FK 없음**. 대화 이력 삭제·세션 삭제가 감사에 영향 없음.
4. **접근 통제** — 감사 조회는 감사 권한자만(v0 최소 구현, 자리 확보).

## 불변식
- `audit_log`에 FK가 없다.
- 세션/이력 삭제 후에도 해당 감사 레코드가 존재한다.
- 감사 기록 실패가 사용자 응답을 막지 않는다(단, 경고 로그로 가시화).
- trace_id로 요청↔감사↔로그를 추적할 수 있다.

## 테스트 시나리오 (TDD)
- [ ] 채팅 1회 → 감사 레코드 1건 생성(trace_id·requester·질문·응답·근거).
- [ ] 세션 삭제 후에도 감사 레코드 유지.
- [ ] 감사 저장 실패 주입 → 사용자 응답은 정상, 경고 로그 남음.
- [ ] 기록 범위 설정을 '최소'로 → 전문 대신 최소 필드만 기록(스키마 변경 없이).
- [ ] audit_log 스키마에 FK 제약이 없음(구조 검증).

## 하지 않을 것 (v0)
감사 대시보드 UI, 보관 기간 자동 정책, 익명화 파이프라인. — B 정책 결정 소관.

# llmhub 팀 온보딩 — 이게 뭐고, 왜 이렇게 만들었나

> 이 문서는 **프로젝트를 처음 보는 사람**이 혼자 읽고 전체 그림을 잡도록 쓴 것이다. 사전 지식은 가정하지 않는다.
> 지금까지 v0를 혼자 만들어 끝냈고, 이제 팀에 공개해 함께 논의하려 한다. 코드를 파기 전에 이걸 먼저 읽으면 좋다.

---

## 1. 한 줄로

**사내 문서에서 근거를 찾아, 그 근거를 대며 답하는 채팅 시스템.** 로그인·권한·감사가 처음부터 들어가 있고,
특정 회사/업무에 종속되지 않게 만들었다.

조금 풀면 — 회사가 가진 규정·매뉴얼·문서를 넣어두면, 직원이 챗봇에게 물어보고 **"이 문서 이 부분에 따르면 …"**
식으로 출처가 붙은 답을 받는다. 아무나 모든 문서를 보는 게 아니라 **권한(역할)에 따라 볼 수 있는 문서만** 검색되고,
누가 무엇을 물었고 무슨 답이 나갔는지 **감사 기록**이 남는다.

### 왜 이런 걸 만드나

- **"근거 없는 챗봇"의 문제.** 일반 LLM 챗봇은 그럴듯하게 지어낸다(환각). 사내 업무에선 위험하다. 그래서 이 시스템은
  **먼저 사내 문서를 실제로 검색**하고, 찾은 조각만 근거로 답한다. 답보다 **근거가 먼저** 화면에 뜬다.
- **권한과 감사가 나중에 붙이기 어렵다.** RAG 데모는 많지만, "이 사람은 이 문서만 볼 수 있다"·"기록이 남는다"를
  뒤늦게 얹으면 구조가 흔들린다. 그래서 **처음부터** 인증·접근 권한·감사를 뼈대에 넣었다.
- **회사마다 설치해 쓰는 제품.** SaaS 한 곳에 모으는 게 아니라, **회사별로 자기 인프라에 설치**(on-prem)하는 걸
  전제로 한다. 문서·대화가 회사 밖으로 안 나간다.

### 두 가지 정체성 키워드

- **도메인 중립.** 코어(핵심 엔진)에는 "규정"·"인사"·"ERP" 같은 특정 업무 개념을 넣지 않는다. 어느 회사, 어느 문서
  종류에도 쓸 수 있는 **범용 문서-검색-채팅 런타임**이다. 업무 특화는 위에 얹는 것이지 코어에 박는 게 아니다.
- **도메인 중립 ≠ 언어 중립.** 코어는 업무를 모르지만, **언어는 설치 설정**이다. v0는 한국어(검색에 한국어
  형태소 분석기 `nori` 사용). 언어 특화 부품은 **교체 가능**하게 뒀다.

참고 모델은 **Open WebUI**다. 방향은 비슷하되(문서 기반 채팅), 기능을 그대로 복제하는 게 아니라 **Spring 생태계 +
권한·감사 내장 + 설치형**이라는 우리 색으로 다시 지었다.

## 2. v0가 실제로 하는 일 (처음부터 끝까지)

한 번의 질문이 시스템을 어떻게 지나가는지 따라가 보자.

1. **문서 넣기(색인).** 관리자가 색인 API로 문서를 올린다(PDF·txt·hwp). 시스템은 텍스트를 추출하고(Tika/hwplib),
   **적당한 크기 조각으로 자르고**(청킹), 각 조각을 **벡터로 바꿔**(임베딩) 검색엔진(Elasticsearch)에 저장한다.
   각 조각에는 어떤 문서인지·어디 위치인지·**누가 볼 수 있는지(접근 태그)**·언제·어떤 임베딩 모델로 만들었는지가
   함께 저장된다. 원본 파일도 따로 보관한다(나중에 다시 색인하려고).
2. **로그인하고, 권한이 정해진다.** 사용자가 Keycloak으로 로그인한다. 요청이 코어에 닿기 전, **앞단 게이트**가
   그 사람의 역할(USER/ADMIN)을 보고 **볼 수 있는 문서 태그 집합**을 확정한다. 예: USER → `{public}`,
   ADMIN → `{public, restricted}`. 이 확정은 여기서 **끝난다** — 뒤 단계는 다시 판단하지 않는다.
3. **질문하고, 검색하고, 답을 만든다.** 사용자가 대화창에서 질문한다. 시스템은 최근 몇 턴의 대화를 컨텍스트로 챙기고,
   **아까 확정된 접근 태그로 필터한** 하이브리드 검색(키워드 BM25 + 의미 벡터)을 돌려 관련 문서 조각을 찾는다.
   그 조각들을 프롬프트에 넣어 LLM에게 "이 근거만 보고 답하라"고 시킨다.
4. **답이 스트리밍으로 온다.** 응답은 실시간 조각으로 흐른다. **먼저 `근거`(sources)** — 실제 검색된 문서 조각 —
   가 확정돼 화면에 뜨고, 그 다음 LLM이 만든 **답 텍스트**가 한 글자씩 흘러나온다. 장애가 나면 깔끔한 `오류`로 닫히고
   사용자가 상황을 안다.
5. **감사 기록이 남는다.** 대화가 끝나면(또는 실패·취소돼도) **감사 로그**에 한 건이 남는다 — 누가·언제·무엇을
   물었고·무슨 근거로·어떻게 끝났는지(완료/취소/오류). 이 기록은 **대화 이력과 완전히 별개**라, 사용자가 자기 대화를
   지워도 감사는 남는다.

이 흐름이 end-to-end로 도는 것이 **v0의 목표**였고, 지금 다 된다. 구체적인 "v0 완성 체크리스트"(13개 항목,
전부 통과)는 [`docs/00-overview.md`](../00-overview.md)에 있다.

## 3. 아키텍처 한눈에

요청은 위에서 아래로 흐른다.

```
CLIENT(Next.js)  →  EDGE(Spring Security)  →  CORE(WebFlux+Spring AI)  →  DATA·RAG(PG·ES)  →  INFRA(LiteLLM)
```

- **CLIENT** — 사용자가 쓰는 웹 UI(Next.js). 채팅 화면·세션 목록. 여기서 중요한 게 **BFF**(Backend-For-Frontend):
  브라우저는 코어를 직접 부르지 않고 **오직 이 BFF만** 부른다. BFF가 코어의 스트림을 UI가 이해하는 형식으로 번역하고,
  토큰을 헤더로 바꿔 코어에 넘긴다. 코어·검색엔진·DB·LLM 게이트웨이는 **내부망 전용**이라 밖에서 안 보인다.
- **EDGE** — Spring Security 필터체인. `CORS → JWT 검증 → RBAC(역할→접근태그 확정) → RateLimit` 순서.
  여기의 **RBAC 단계가 "앞단 게이트"** — 권한을 여기서 확정한다.
- **CORE** — 실제 채팅 오케스트레이션(Spring WebFlux 논블로킹 + Spring AI). 검색을 명시적으로 부르고, 근거를
  프롬프트에 넣고, LLM을 스트리밍 호출한다.
- **DATA·RAG** — **PostgreSQL**(사용자·세션·대화이력·감사) + **Elasticsearch**(문서 조각의 벡터+키워드 검색,
  한국어 nori) + 원본 파일 저장.
- **INFRA** — **LiteLLM 게이트웨이**. 코어는 "모델 이름"만 넘기고, 실제 어느 모델(온프레 Ollama / 클라우드
  OpenAI·Anthropic·Google)로 보낼지·페일오버는 게이트웨이가 정한다.

### 모듈 지도

시스템을 5개 모듈로 나눴고, 각각 독립적으로 테스트된다. 모듈끼리는 **인터페이스(계약)로만** 통신한다.

| 모듈 | 하는 일 |
|---|---|
| **IDX** 색인 | 업로드→추출→청킹→임베딩→저장, 원본 보관, 재색인 |
| **AUTH** 인증·게이트 | Keycloak 검증, 역할→접근태그 확정(앞단 게이트) |
| **SEARCH** 검색 | 하이브리드 검색(BM25+벡터), 접근태그 필터, 근거 생성 |
| **CHAT** 채팅 | LLM 오케스트레이션, 이벤트 스트리밍, 세션·이력 |
| **AUDIT** 감사 | 감사 로그(독립 수명주기) |

의존 방향: `CHAT → (AUTH가 확정한 태그 소비) → SEARCH → 근거 → 응답`. AUDIT는 응답과 독립으로 비동기 기록.
IDX는 관리자 경로로 독립. 더 자세한 계층·인터페이스 계약은 [`docs/01-architecture.md`](../01-architecture.md).

### 기술 스택

| 계층 | 기술 |
|---|---|
| Client | Next.js(App Router) · Vercel AI SDK(`useChat`) · zustand · next-auth(Keycloak) |
| Edge | Spring Security 필터체인 · Keycloak(OIDC) |
| Core | Spring Boot **WebFlux** · Spring AI ChatClient |
| Data | PostgreSQL(JPA/Hibernate·Flyway) · Elasticsearch(dense_vector + BM25, nori) |
| ETL | Apache Tika · hwplib · 토큰 크기 청킹(글자 경계 안전) · EmbeddingModel(게이트웨이 경유) |
| Infra | LiteLLM Gateway · Ollama / OpenAI · Anthropic · Google |

## 4. 왜 이렇게 설계했나 (핵심 결정들)

코드를 읽다 "왜 이렇게 했지?" 싶은 지점들의 이유다. 이 결정들은 [`docs/02-decisions.md`](../02-decisions.md)에
`S`(결정)·`E`(경계)로 못박혀 있고, 구현이 반드시 지켜야 하는 제약이다.

- **근거는 서버 검색 결과에서만 나온다 (S6).** LLM 출력에서 근거를 파싱하지 않는다. `sources`는 서버가 실제로 검색한
  조각으로 서버가 만든다. 그래서 검색을 RAG Advisor 안에 숨기지 않고 **CHAT이 명시적으로 먼저 부른다** — 그래야
  근거를 첫 토큰보다 먼저 확정 발행할 수 있고, "지어낸 근거"가 원천 차단된다.
- **권한은 구조로 강제한다 — 앞단 게이트 (S4).** 접근 태그 확정은 오직 필터체인의 RBAC 단계에서. 뒤의 검색·RAG는
  태그를 **소비만** 하고 다시 판단하지 않는다. 컨트롤러가 권한을 확인하는 구조라면 새 엔드포인트가 그걸 빼먹을 수 있어서,
  **빼먹을 경로 자체를 없앴다.**
- **감사는 대화와 독립이다 (S5).** 감사 로그는 대화 이력과 **다른 테이블, FK 없음**. FK가 있으면 사용자가 대화를
  지울 때 감사도 지워지거나 삭제가 막힌다. 감사는 사용자 삭제와 무관하게 남아야 하므로 값-복사로 저장한다.
- **색인·검색 임베딩 모델은 같아야 한다 (S8-4).** 다르면 에러 없이 검색 품질만 조용히 무너진다. 그래서 임베딩 모델을
  설정으로 고정하고 각 조각 메타에 기록한다. 모델이 바뀌면 재색인, 차원이 바뀌면 새 인덱스가 필요하다.
- **WebFlux(논블로킹)와 JPA(블로킹)의 공존.** 코어는 스트리밍을 위해 논블로킹(WebFlux)인데 DB는 블로킹(JPA)이다.
  블로킹 DB 접근이 이벤트 루프를 막으면 스트리밍이 굶는다. 그래서 **블로킹 접근은 전용 격리 스케줄러**로만 넘긴다.
  이게 실제로 지켜지는지 테스트로 강제(BlockHound·부하 격리 테스트)한다.
- **BFF가 프로토콜을 번역한다.** 코어는 S6가 요구하는 명명된 이벤트(`text`/`sources`/`error`/`done`)를 그대로 내고,
  브라우저의 AI SDK가 이해하는 파트(`text-delta`/`data-sources`/…)로의 번역은 **BFF 한 곳**에만 둔다.
- **한국어 글자 경계 청킹.** 흔한 토큰 분할기는 바이트 BPE 토큰을 잘라 디코딩해서, 한 글자가 여러 토큰에 걸치는
  한국어에서 조각 경계가 글자 중간을 잘라 깨진 문자(U+FFFD)를 남긴다. 그래서 토큰은 **크기 재는 데만** 쓰고, 자르는
  위치는 문장→단어→코드포인트 순으로 찾는다.
- **설정 외부화 (REL-4).** 모델·분석기·상한·타임아웃 등 배포마다 다른 값은 코드에 박지 않고 환경변수로 뺀다.
  "회사별 설치형"이라 같은 산출물이 다른 회사에서 다른 설정으로 돌아야 한다.

즉 "뼈대가 end-to-end로 돌고, 검증까지 된" 상태다. 프로덕션 실배포 마감(리버스 프록시·실 도메인 등)과 기능 확장은
아직 열려 있다.

## 6. 직접 돌려보기

세 줄 요약:

```bash
git config core.hooksPath .githooks && cp .env.example .env   # 값 채우기
docker compose up -d && ./docker/keycloak/bootstrap-dev.sh     # 인프라 + dev 계정
# 코어: cd backend && ./gradlew --no-daemon bootRun (:8080) / BFF: cd frontend && npm i && npm run dev (:3000)
```

로그인은 `dev-user`(USER) 또는 `dev-admin`(ADMIN). 실제 색인·질문을 돌리려면 호스트 Ollama에 임베딩·채팅 모델을
받아둬야 한다. 자세한 설정·운영 배포·장애 추적은 루트 [`README.md`](../../README.md)에 다 있다.

## 7. 어떻게 일하는가 (이 저장소의 운영 방식)

이 프로젝트는 규율을 좀 특이하게 잡았다. 팀으로 일하려면 이 부분을 이해하는 게 중요하다.

- **[`CLAUDE.md`](../../CLAUDE.md)가 "헌법"이다.** 절대 규칙·확정 스택·워크플로우가 여기 있고, 모든 작업의 최상위
  규칙이다. 규칙이 충돌하면 우선순위: **보안 > 데이터 정합성 > 결정 준수 > 테스트 > 기능 > 속도.**
- **`docs/`는 에이전트용, `docs/syj/`는 사람용.** `docs/00`~`08`·`requirements`·`reviews`·`LEARNINGS`·
  `OPEN-QUESTIONS`는 (사람도 읽지만) 주로 **AI 에이전트가 규칙·계약·이력을 참조**하는 거버넌스·지식 문서다.
  지금 읽는 이 `docs/syj/`는 **사용자와 주고받는 공간**이다([`docs/syj/README.md`](README.md) 참고).
- **테스트 없이 기능을 병합하지 않는다(TDD).** 실패하는 테스트를 먼저 쓰고, 그게 빨간불인지 확인하고, 최소 구현으로
  초록불을 만든다. 리뷰 문화도 "추측 금지, 코드로 확인"이라 발견마다 `[실증]`/`[코드]`/`[미검증]`으로 근거를 표시한다.
- **비밀정보는 커밋하지 않는다(SEC-3).** 키·토큰·비밀번호는 `.env`(gitignore)에. `.env.example`만 커밋한다.
- **커밋 훅이 이 규율을 강제한다.** `.githooks/pre-commit`이 테스트를 돌리고, 테스트 무력화·비밀정보·읽기 전용 문서
  변경을 차단한다.

이 방식 덕에 지금까지 혼자서도 일관되게 진행할 수 있었다. 팀에 공개하면서 이 규율을 유지할지·완화할지도 논의 대상이다.

## 8. 앞으로 / 함께 논의할 것

v0 뼈대는 끝났으니, 이제 **어디로 갈지**가 팀 논의 주제다. 후보들:

- **배포 마감.** 검증된 운영 스택을 실제 회사 호스트에 올리는 나머지: 리버스 프록시(TLS 종료), 실 도메인
  설정, 모니터링. 지금은 compose가 "운영 기준"까지 와 있다.
- **v1 기능 — 무엇을 먼저 열까.** [`CLAUDE.md`](../../CLAUDE.md) §7이 "지금 하지 말 것"으로 미뤄둔 것들 중 하나를
  골라 스코프를 연다: 답변 마크다운 렌더 · 세션 이름 변경 · 이력에 근거 표시 · 관리자 색인 UI · 쿼리 재작성 ·
  판단 게이트 · 능동 알림 등. **자리는 열어뒀지만 구현은 안 한** 것들이라, 여는 순간이 제품 방향 결정이다.
- **관측성.** 지금은 추적 ID 기반 로깅·감사는 있으나 메트릭/대시보드는 없다.
- **OQ-011.** 보안을 위해 Keycloak `Revoke Refresh Token`을 켜기로 하면, 그때 동시 토큰 갱신 경쟁을 realm 설정
  (재사용 grace)으로 흡수한다.

각 항목의 "확장 경계"(어디까지 열려 있고 무엇을 조심해야 하는지)는 [`docs/02-decisions.md`](../02-decisions.md)의
`E`(경계) 항목에 적혀 있다. **v0의 핵심 약속은 "지금 안 만들되, 나중에 못 만들게 되진 않는다"** — 그래서 무엇을
열든 뼈대를 다시 갈아엎을 필요는 없다.

# 프로젝트 전반 평가 — 방향성·구조·관리 체계 (2026-07-13)

> **평가 방식.** 이 보고서는 코드를 한 줄씩 정독하는 리뷰가 아니라, 여섯 개 차원(아키텍처·설계 결정 / 거버넌스·지침 체계 / 문서 정합성 / Fresh-eyes 새 팀원 시뮬레이션 / v0 완결성·누락 / 프로세스 실증)을 각각 독립적으로 병렬 평가한 뒤, 모든 발견에 대해 별도의 변호(defense) 패스 — 근거를 재확인하고 이미 기록된 유보와 혼동하지 않았는지 검증하는 절차 — 를 거쳐 `valid`(유효)/`deferred`(이미 기록된 유보)/`rejected`(기각)로 판정하고 종합한 결과다. 총 27건의 발견 중 21건이 `valid`, 6건이 `deferred`, 0건이 `rejected`로 판정되었다. 목적은 "이 프로젝트가 팀에 공개될 준비가 되었는가"라는 큰 줄기를 판단하는 것이며, 세부 구현 정확성 검증이 아니다.


> **패널 구성과 지휘 검증.** 평가자 6·변호 6·종합 1 총 13개 에이전트(전부 Sonnet 5, 읽기 전용)가 병렬로 수행했고,
> 지휘(메인 세션, Fable 5)가 high 등급 발견 5건을 독립적으로 재검증했다:
> GOV-1(훅의 체크박스 정규식 `grep -vE '^[+-][[:space:]]*- \[( |x)\] '`은 새 체크박스 줄 **추가**를 통과시킴 — 훅 파일 직접 확인 [코드]) ·
> GOV-2(`.github/` 디렉토리 부재 직접 확인 [실증]) ·
> GOV-3(`.githooks/pre-commit:106-124`, node_modules 부재 시 검사를 건너뛰고 통과 — 직접 확인 [코드]) ·
> V0-1(백업 관련 문서 grep 결과 런북 0건 [실증] — LEARNINGS의 snapshot 히트는 무관한 내용) ·
> PROC-1([실증] 독립 계수: 124개 커밋 중 `Tests:` 트레일러 보유 50개 — **최근 UI 커밋들(에이전트 작성)도 누락**, 규칙은 docs/07:24에 "기능 커밋에 필수"로 명시).
> 평가자의 세부 수치(예: PROC-1의 '48개')는 기능 커밋만 센 각자의 방법론에 따른 것으로, 방향성은 지휘 계수와 일치한다.

## 총평

llmhub는 1인 개발자(syj)와 에이전트가 함께 만든 프로젝트치고 이례적으로 성숙한 거버넌스 체계를 갖췄다. CLAUDE.md·docs/01~08로 이어지는 결정 문서 체계가 단순한 규정집이 아니라 ArchUnit 테스트, WebFilter 순서, BlockHound 계측, pre-commit 훅 같은 **실행 가능한 강제 장치**로 코드에 반영되어 있고, 리뷰에서 나온 발견 대부분이 실제로 후속조치(수정·반려·유보)되었다는 점이 커밋 이력으로 확인된다. S4(앞단 게이트)·S6(서버 근거)·S8-4(임베딩 모델 고정) 같은 핵심 결정이 문서상의 선언에 그치지 않고 실제 호출 순서와 필터 체인으로 구조화되어 있다는 것은 이 규모의 프로젝트에서 흔치 않은 성취다.

동시에, 이 성숙도는 "1인이 매 순간 훈련된 상태로 지켰다"는 전제 위에 서 있다는 한계도 뚜렷하다. 팀 전환을 앞두고 볼 때 세 가지가 걸린다. 첫째, 규율을 강제하는 장치 자체(체크박스 잠금 훅, pre-commit 훅)에 실제로 우회 가능한 허점이 있고, 이를 서버 측(CI)에서 이중으로 막는 장치가 전무하다 — 로컬 훅 하나에 전적으로 의존하는 구조는 팀 규모에서 위험하다. 둘째, 문서가 스스로 규정한 규율(예: `Tests: N passed` 트레일러)이 프로젝트 후반 약 38%의 커밋에서 조용히 무너졌는데 이것이 어디에도 기록되지 않았다 — "문서를 읽으면 실제로 지켜지고 있다고 오인하게 되는" 지점이 존재한다는 뜻이다. 셋째, 백업·복구, 시크릿 로테이션, 라이선스 표기처럼 설치형 제품을 실제 고객에게 넘길 때 필요한 운영 기본기 몇 가지가 조용히 비어 있다. 종합하면, **핵심 아키텍처·보안 경계는 신뢰할 만한 수준이나, 거버넌스 장치의 견고성과 운영 준비도는 팀 공개 전에 반드시 손볼 필요가 있다.**

## 최대 강점 (3~5)

1. **S4/S6 같은 핵심 결정이 구조로 강제되어 있다.** 접근태그는 SEARCH/CHAT보다 앞선 WebFilter 단계(`AccessTagGateWebFilter`)에서만 확정되고 하위 계층은 소비만 한다. 근거(sources)는 LLM 호출 전에 먼저 검색을 실행하고 `Sources` 이벤트를 `text`보다 먼저 발행하는 실제 호출 순서(`ChatService.java:107-117`)로 구현되어 있어, "LLM 출력에서 근거를 파싱하지 않는다"는 원칙이 우회 불가능한 형태로 박혀 있다.
2. **모듈 경계와 논블로킹/블로킹 공존이 빌드 시점 테스트로 검증된다.** `ModuleBoundaryTest.java`의 ArchUnit 규칙이 IDX/AUTH/SEARCH/AUDIT 상호 격리와 common의 무지(無知)를 강제하고, `BlockHoundArmedTest.java`·`LoadIsolationTest.java`가 이벤트 루프에서의 블로킹 누수를 실제로 실패시킨다. 문서상의 규칙이 아니라 실행 가능한 규칙이다.
3. **리뷰→발견→수정의 폐루프가 커밋 이력으로 실증된다.** R-1~R17·L-1~L12·F1~F7·B1~B3 등 리뷰 발견 대부분이 수정되거나(예: F1 감사 유실을 `ChatService.java:101-103`에서 직접 수정), 근거를 들어 반려되거나(LEARNINGS), 사람 확인 후 유보(OQ)되었다. 현재 진짜 OPEN 상태인 항목은 13개 OQ 중 1건(OQ-011)뿐이다.
4. **CLAUDE.md의 우선순위가 자율 진행 프로토콜의 실제 판정 로직과 정확히 대응한다.** 보안>정합성>결정준수>테스트>기능>속도라는 선언이 docs/08의 10단계 HALT/SKIP 판정과 문서 자체에서 명시적으로 매핑되어 있고, OPEN-QUESTIONS.md의 반증 가능한 보류 선언 문구("해당 없음: 도달 가능한 보안 사안·S 항목·데이터 정합성·결정 문서")가 13개 항목 전부에 문자 그대로 반복된다.
5. **하이브리드 검색의 비자명한 함정(kNN pre-filter vs bool post-filter)을 팀이 스스로 발견해 고쳤다.** SEC-2("권한 없는 조각은 검색 단계에서 배제")를 문자 그대로 지키기 위해 `LinearCombinationMerger.java:13-18`에서 필터 적용 위치의 차이를 코드 주석과 LEARNINGS.md에 남기며 교정한 흔적이 남아 있다.

## 최대 리스크 (3~5)

- **[GOV-2, high]** 테스트·비밀정보·문서잠금 강제가 전부 로컬 `pre-commit` 훅 하나에 의존하는데, 이를 켜는 `core.hooksPath`는 저장소에 커밋되지 않는 클론별 로컬 설정이고 이를 자동 설치·검증하는 CI가 전혀 없다. `.github/` 디렉터리 자체가 없고, `frontend/package.json`에 훅 자동설치 스크립트도 없다. 신규 클론이나 `--no-verify` 앞에서 이 모든 규율이 무력화될 수 있다.
- **[GOV-1, high]** 체크박스 잠금 훅(`.githooks/pre-commit:72-79`)이 실제로는 "체크박스 마커로 시작하는 아무 줄이나 허용"하는 정규식만 쓰고 있어, 완전히 새로운 시나리오 줄 추가나 기존 문구 전체 교체가 조용히 통과한다. 재현 시뮬레이션으로 실제 우회 가능함이 확인되었다. 이 잠금이 지키려는 "에이전트가 스스로 스코프를 넓히지 못하게"라는 목적 자체가 형식만 지키면 뚫린다.
- **[GOV-3, high]** 프론트엔드 pre-commit 검사는 `frontend/node_modules`가 없으면(신규 클론 기본 상태) 타입검사·테스트·빌드를 전부 건너뛰고 그대로 커밋을 통과시킨다 — 실패가 아니라 무검증 통과다. CLAUDE.md 절대 규칙 7("테스트 없이 기능 코드를 병합하지 않는다")과 정면으로 충돌한다.
- **[V0-1, high]** 백업·복구 절차가 저장소 어디에도 문서화되어 있지 않다. PostgreSQL·Elasticsearch·업로드 원본이 전부 호스트 바인드마운트뿐이고 pg_dump/ES snapshot/복구 런북이 전무하며, 로드맵 문서에도 언급이 없는 "조용한 누락"이다.
- **[PROC-1, high]** docs/07이 프로젝트 시작부터 못박은 `Tests: N passed` 커밋 트레일러 규칙이 문서상으로는 한 글자도 안 바뀐 채, 실제로는 전체 124개 커밋 중 후반 약 38%(48개, F1~F7/B1~B3 보안·감사 수정 및 UI 리디자인 전체 포함)에서 아무 기록 없이 조용히 폐기되었다. 문서만 읽으면 규율이 지켜지고 있다고 오인하게 된다.

## 우선순위 권고

### 팀 공개 전 필수

- 체크박스 잠금 훅을 줄 단위 독립 매칭에서 삭제줄-추가줄 페어링 검증으로 교체한다 (**GOV-1**).
- 동일 검사를 수행하는 서버측 CI(GitHub Actions 등)를 추가하고 브랜치 보호 규칙을 걸어, 로컬 훅 미설치·`--no-verify`에도 막히게 한다 (**GOV-2**).
- 프론트엔드 pre-commit 훅이 `node_modules` 부재 시 검사를 건너뛰지 말고 실패시키거나 자동 설치 후 검사하도록 고친다 (**GOV-3**).
- 거버넌스 문서 편집 승인(`LLMHUB_DOC_EDIT_APPROVED`)에 신원이 남는 절차(코드오너·PR 승인)를 도입한다 (**GOV-4**, 다만 팀 논의 대상으로 이미 인지·기록됨).
- `docs/01`의 "모듈 간 통신은 인터페이스로만" 규칙과 `SearchService`(concrete class) 사이의 불일치를 정리한다 (**ARCH-1**).
- README의 문서 표에 `docs/06`·`docs/07`·`docs/reviews/`·`docs/ui-design.md`를 추가한다 — 문서 정합성 차원과 Fresh-eyes 차원이 독립적으로 동일 결함을 발견했다 (**DOC-1**).
- 색인 API(`POST /api/index`) 호출 예시(curl + 토큰 획득 방법) 또는 seed 스크립트를 온보딩 문서에 추가해 "직접 돌려보기"가 실제로 근거 있는 답까지 끝까지 되도록 한다 (**ONBOARD-1**).
- 백업/복구 절차를 최소 수준(pg_dump/ES snapshot + 복구 런북)으로 채우거나, 즉시 OPEN-QUESTIONS.md에 의식적 유보로 등록한다 (**V0-1**).
- `docs/07`의 `Tests: N passed` 트레일러 규칙을 실제 관행에 맞춰 사람 승인으로 재정합하거나, 훅에 트레일러 존재 검증을 추가한다 (**PROC-1**).
- 프론트 `npm test` 범위가 `lib/**`에만 한정되어 `auth.ts`·React 컴포넌트가 구조적으로 미검증 상태인 문제를 OPEN-QUESTIONS.md에 정식 등록한다 — 두 차례 리뷰(R-4, F3/F4)가 이미 지적했으나 추적 기록이 없다 (**PROC-2**).
- pre-commit 훅의 4.5분 전체 스위트 게이트가 "main 직접 커밋" 1인 체제 전제로 설계되어 있다 — 팀 전환 시 브랜치/PR/CI 이관 계획을 `docs/07`에 명문화한다 (**PROC-4**).

### 나중·선택

- IDX의 스트라이프 락(64칸, 인프로세스)은 "회사별 단일 인스턴스" 전제에서만 유효함이 코드 주석에 명시되어 있다 — 다중 인스턴스 배포가 로드맵에 오르면 그때 PG advisory lock 전환을 계획한다.
- `docs/06`의 "관련 docs도 함께 갱신" 문구에 `docs/08` §B 절차 참조를 덧붙여 문서 간 문구 상충 인상을 없앤다 (**GOV-7**).
- ES 재색인 런북을 `.env.example` 주석에서 운영 문서로 승격한다 (**V0-3**, REQ-IDX 문서에 이미 "자리만 열어둠"으로 명시된 v0 스코프 결정).
- 계정 수명주기(퇴사자 처리, **V0-5**)·지원 브라우저 범위(**V0-7**)를 각각 한두 문단으로 문서화한다.
- pre-commit 훅을 변경된 모듈만 선별 테스트하도록 최적화해 팀 확장 시 대기 시간 마찰을 줄인다.

## 차원별 상세

### 아키텍처·설계 결정 (ARCH)

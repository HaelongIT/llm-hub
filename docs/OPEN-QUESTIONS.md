# OPEN QUESTIONS — 보류된 질문·작업 로그

> 무중단 진행 중 **지금 멈출 이유는 없지만 나중에 확인해야 하는 것**을 남긴다.
> 절차는 `docs/08-autonomy-protocol.md`. 세션 시작 시 OPEN 항목을 읽는다.

## LEARNINGS.md와의 경계 (한 줄)

**아직 하거나 정해야 할 일이 남았으면 OPEN-QUESTION, 남은 게 없는 확정된 사실이면 LEARNING.**
둘에 동시에 있지 않는다. OQ가 닫히면서 지식이 남으면 그 사실을 `LEARNINGS.md`로 옮긴다.

## 규칙

- **ID:** `OQ-NNN`. 단조 증가, 재사용 금지.
- **상태:** `OPEN` → `ANSWERED`(사람이 결정, 아직 미구현) → `RESOLVED`(구현 완료, 커밋 sha와 커버 테스트를 인용) · `WONTFIX`(**사람만** 설정) · `BLOCKED`(모듈 의존, WIP 상한에 포함 안 됨)
  - 에이전트는 `OPEN → RESOLVED`만 스스로 옮길 수 있고, 그때 커밋 sha와 통과 테스트를 인용해야 한다.
  - `ANSWERED`와 `WONTFIX`는 사람만 설정한다. 항목을 **삭제하지 않는다.** 상태만 전이한다.
- **WIP 상한 10건**(7건에서 경고). 도달하면 다음 시나리오를 시작할 수 없다 → HALT하고 사람에게 정리를 요청한다.
- **자동 차단 태그** `SEC-REACHABLE` · `S-ITEM` · `DATA` · `AUTHZ` · `AUDIT` · `EMBED-MODEL` 은 **OPEN 상태로 존재할 수 없다.** 이런 항목은 보류가 아니라 HALT다.
- **`Revisit-trigger`는 필수다.** 없으면 무효 항목이다. 7일 넘은 OPEN 항목은 상태 보고 맨 위에 뜬다.
- 모든 항목의 `판단 근거`에는 다음 문장이 문자 그대로 들어가야 한다. 정직하게 쓸 수 없으면 보류할 수 없다.
  > `해당 없음: 도달 가능한 보안 사안 · S 항목 · 데이터 정합성 · 결정 문서`

**현재 OPEN: 0건 / 10건** · BLOCKED 0건 · ANSWERED 3건(OQ-006, OQ-007, OQ-008)

## OQ-001 · `LocalFileStorage.read(key)`가 키를 검증하지 않는다

- **태그:** `TEST-GAP` `SEC-WATCH`
- **상태:** **RESOLVED** (2026-07-10)
- **연 시각:** 2026-07-10 (커밋 `54aa1de`)
- **Revisit-trigger:** 다음 IDX 사이클 (미루지 않는다. 자체 커밋으로 즉시 처리)
- **Owner:** agent

**상황:** `read(String key)`가 키를 그대로 `root.resolve(key)`에 넘긴다.

**발견:** `..`나 절대경로, 빈 문자열이 키로 들어오면 원리상 저장 루트를 벗어난 파일을 읽을 수 있다. 이를 막는 테스트가 없다.

**판단 근거:** 충족한 CONTINUE 조건 — *잠재 하드닝인데 현재 입력으로 도달 불가이고, 수정이 순수 추가형 가드 + 테스트다.* 호출자를 grep한 결과 `read()`의 **프로덕션 호출자가 0개**이고(테스트 한 곳뿐), 설계상 키의 출처는 서버가 생성해 `document.original_path`에 저장한 UUID다. 클라이언트가 키에 영향을 줄 경로가 현재 존재하지 않는다. 수정에 계약 변경이나 사람의 판단이 필요 없다. 이것은 *결정*이 아니라 끝나지 않은 *작업*이므로 보류하고 즉시 일정에 넣는다.

> 해당 없음: 도달 가능한 보안 사안 · S 항목 · 데이터 정합성 · 결정 문서

**트립와이어:** 클라이언트 입력이 `key`에 영향을 줄 수 있는 호출자가 생기는 즉시 `SEC-REACHABLE`로 재분류하고 **HALT**한다.

**회의적인 리뷰어에게:** 리뷰어는 "보안 문제를 SEC-WATCH로 이름만 바꿔 미룬 것 아니냐"고 지적할 것이다. 답은 — 호출자가 0개라 오늘 악용 경로가 존재하지 않고, 되돌리기 비용이 없는 추가형 가드이며, 다음 사이클에 자체 커밋으로 처리하고 트립와이어를 걸어두었다는 것이다. "나중에"가 아니라 "다음 사이클"이다.

**제안하는 조치:** `..`·절대경로·빈 키·경로 구분자가 든 키를 `resolve` 이전에 거부하는 테스트를 먼저 쓰고, 키가 안전한 단일 경로 세그먼트인지 검증한다.

**해결:** 커밋 `4dd6ff1` — `LocalFileStorage.requireSafeKey()`가 `..`·경로 구분자·빈 키를 파일시스템에 닿기 전에 거부한다. 커버 테스트: `LocalFileStorageTest.오염된_키를_거부한다`(6 케이스) + `정상_키는_읽힌다`. 약속대로 다음 사이클에 자체 커밋으로 처리했고, 트립와이어는 이제 불필요하다.

## OQ-002 · `docker compose up` 전체 기동을 검증하지 않았다

- **태그:** `INFRA` `HUMAN-VERIFY`
- **상태:** **RESOLVED** (2026-07-10)
- **연 시각:** 2026-07-10 (커밋 `b176d54`)
- **Revisit-trigger:** 첫 CHAT end-to-end 시나리오 이전, 또는 릴리스 태그 이전
- **Owner:** human (에이전트에게 compose를 띄울 런타임이 없다)

**상황:** 스택 전체를 한 번도 함께 띄우지 않았다. ES 이미지만 빌드해 `elasticsearch-plugin list`로 nori 설치를 확인했다.

**발견:** compose 그래프(PostgreSQL + Elasticsearch + Keycloak + LiteLLM + 백엔드)가 실제로 기동하고 서로 연결된다는 증거가 없다. `docker compose config`로 문법만, realm JSON은 파서로 문법만 확인했다. Keycloak realm import와 LiteLLM config는 각각 AUTH·CHAT 착수 시점에 실제로 부딪힌다.

**판단 근거:** 충족한 CONTINUE 조건 — *검증에 에이전트가 못 쓰는 런타임이 필요하고 단위·슬라이스 테스트는 green이다.* 현재 TDD 시나리오는 모두 단위·슬라이스 수준이고, 어떤 REQ 시나리오도 전체 스택 기동을 단언하지 않는다. ES 통합 테스트는 Testcontainers가 컨테이너를 직접 띄우므로 compose와 무관하다.

> 해당 없음: 도달 가능한 보안 사안 · S 항목 · 데이터 정합성 · 결정 문서

**회의적인 리뷰어에게:** 리뷰어는 "REL-5(배포 재현성)가 NFR인데 검증을 미뤘다"고 지적할 것이다. 답은 — REL-5는 v0 완성 시점의 요구이지 매 사이클의 요구가 아니고, 지금 기동해도 붙일 애플리케이션이 없으며(AUTH·CHAT 미구현), 첫 CHAT e2e 시나리오가 이걸 강제로 소환한다는 것이다.

**제안하는 조치:** 전용 인프라 점검 — `docker compose up` → 헬스체크 전부 green → 스모크 요청 1회.

**해결:** 실제로 기동해 확인했다.

- Elasticsearch: `status: green`, `_cat/plugins`에 `analysis-nori` 존재.
- Keycloak: `/realms/llmhub` → HTTP 200. **realm import가 실제로 동작한다.**
- LiteLLM: `/health/liveliness` → HTTP 200. config.yaml이 로드된다.
- PostgreSQL: healthy. 코어를 붙이니 Flyway가 마이그레이션 2개를 적용했고 테이블 6개가 생겼다.
- 코어: Netty로 5.4초 만에 기동. `/api/chat/stream`·`/api/index`·`/api/sessions` 모두 인증 없이는 401.
- 살아 있는 DB에서 `audit_log`의 FK 개수가 0임을 재확인했다(S5).

**발견한 실제 문제 1건:** compose의 PostgreSQL 호스트 포트를 5432로 두면, 로컬에 PostgreSQL이 설치된 환경에서 그 서버가 `0.0.0.0:5432`를 먼저 잡아 호스트에서 나가는 연결이 컨테이너가 아니라 로컬 서버로 간다(Docker는 IPv6만 잡는다). 증상은 `password authentication failed for user "llmhub"`이고, 컨테이너 내부 `psql`은 trust 인증이라 정상으로 보여 오진하기 쉽다. `.env.example`의 기본 포트를 5433으로 바꿨다.

## OQ-003 · `docs/07`은 red 테스트 커밋을 허용하는데 훅이 막는다

- **태그:** `PROCESS`
- **상태:** **RESOLVED** (2026-07-10)
- **연 시각:** 2026-07-10 (커밋 `5fc125d`)
- **Revisit-trigger:** 실패하는 테스트만 먼저 커밋하고 싶어지는 첫 순간, 또는 사람 검토
- **Owner:** human

**상황:** `docs/07-git-workflow.md`는 "테스트만 먼저 커밋해도 좋다(`test:`) — 단 그 커밋은 실패하는 테스트 추가임을 메시지에 명시"를 허용한다.

**발견:** `.githooks/pre-commit`은 백엔드 변경이 스테이징되면 전체 테스트를 돌리고 red면 무조건 차단한다. 즉 `docs/07`이 허용한 red `test:` 커밋을 훅이 물리적으로 막는다. 지금까지의 사이클은 모두 Red → Green을 한 커밋으로 묶어서 부딪히지 않았다.

**판단 근거:** 충족한 CONTINUE 조건 — *동작 보존 리팩터·프로세스 정합성 문제로, 현재 어떤 시나리오도 막고 있지 않다.* 두 규칙이 충돌하지만 실무상 Red+Green을 한 커밋으로 묶는 것이 `docs/07`의 "커밋 시점의 코드는 빌드·테스트가 통과해야 한다"와 더 잘 맞는다. 어느 쪽을 정본으로 삼을지는 사람의 결정이다.

> 해당 없음: 도달 가능한 보안 사안 · S 항목 · 데이터 정합성 · 결정 문서

**회의적인 리뷰어에게:** 리뷰어는 "훅이 문서를 어기고 있는데 왜 안 멈췄냐"고 지적할 것이다. 답은 — 이것이 막고 있는 작업이 하나도 없고(모든 사이클이 Red+Green 묶음 커밋으로 진행됨), 훅을 느슨하게 푸는 쪽은 안전장치를 약화시키므로 사람이 정해야 한다는 것이다.

**제안하는 조치:** 둘 중 하나. (a) `docs/07`에서 red `test:` 커밋 허용 문구를 빼고 Red+Green 묶음 커밋을 정본으로 한다. (b) 훅에 `LLMHUB_RED_TEST=1` 탈출구를 둔다. 저는 (a)를 권한다 — red 커밋은 `main`을 항상 green으로 유지한다는 `docs/07`의 다른 규칙과도 충돌한다.

**해결:** 사람이 (a)를 선택했다. `docs/07`에서 red 커밋 허용 문구를 삭제하고, TDD 사이클 하나를 한 커밋으로 묶는 것을 정본으로 했다.

- 잃는 것(테스트가 실제로 실패했다는 커밋 단위 증거)은 **커밋 본문의 `Red 확인:` 기록**으로 보존한다. 실패 이유가 "미구현"이었음을 적는다.
- `test:` 타입은 살렸다. **green인 테스트만 추가**할 때 쓴다(통합 테스트 보강, e2e).
- `.githooks/pre-commit`은 **동작을 바꾸지 않았다.** 이미 정본대로 막고 있었다. 근거 주석만 붙였고 탈출구는 늘리지 않았다.
- 문서를 고친 뒤 백엔드를 일부러 깨뜨려 red 커밋이 여전히 차단되는지 확인했다.

문서 안의 두 규칙이 서로를 부정하고 있었고, 훅이 조용히 한쪽 편을 들고 있었다. 규칙을 실행 가능한 형태로 옮기지 않았다면 모순은 계속 안 보였을 것이다.

## OQ-004 · "비-ADMIN 업로드 → 403"은 AUTH가 없어 진행할 수 없다

- **태그:** `BLOCKED-BY-MODULE`
- **상태:** **RESOLVED** (2026-07-10)
- **연 시각:** 2026-07-10 (커밋 `ccafb38`)
- **Revisit-trigger:** AUTH 모듈의 필터체인·RoleTagMapper가 들어오는 즉시
- **Owner:** agent

**상황:** `REQ-IDX`의 마지막 미완 시나리오다. IDX의 다른 9개 시나리오는 모두 통과했다.

**발견:** 403을 단언하려면 색인 API 엔드포인트와 Security 필터체인이 있어야 한다. 둘 다 AUTH 모듈의 산출물이다. 지금 만들면 AUTH의 설계를 IDX 쪽에서 앞질러 정하게 된다.

**판단 근거:** 충족한 SKIP 조건 — *아직 만들지 않은 모듈에 의존한다.* 미충족 의존성이 `AUTH`로 이름 지어져 있고, AUTH가 도착하면 자동으로 풀린다. 어려워서 건너뛰는 것이 아니다. 색인 로직 자체는 진입 경로와 분리된 서비스 계층에 이미 있으므로(S1, E1), 엔드포인트를 얹는 일이 색인 로직을 바꾸지 않는다.

> 해당 없음: 도달 가능한 보안 사안 · S 항목 · 데이터 정합성 · 결정 문서

**회의적인 리뷰어에게:** 리뷰어는 "색인 API가 인증 없이 열려 있는 것 아니냐"고 지적할 것이다. 답은 — 색인 API 자체가 아직 존재하지 않는다(컨트롤러가 없다). 노출된 엔드포인트가 없으므로 도달 가능한 보안 구멍이 아니다. AUTH를 만들면서 엔드포인트와 게이트를 함께 얹는다.

**제안하는 조치:** AUTH 착수 후 `POST /api/index`(가칭)에 ADMIN 역할 요구를 걸고, USER 토큰으로 403을 단언한다(SEC-2).

**해결:** 커밋 `efdbad9` — `POST /api/index`를 만들고 필터체인에서 `hasRole("ADMIN")`으로 막았다. 판정을 컨트롤러가 아니라 체인에 둬 권한 검사 누락 경로가 생기지 않게 했다. 커버 테스트: `IndexControllerTest`(ADMIN 200 / USER 403 / 무인증 401 + 블로킹 격리 스레드 단언).

## OQ-005 · "부하 시에도 블로킹 접근이 논블로킹 흐름을 막지 않는다"를 부하로 검증하지 않았다

- **태그:** `PERF` `HUMAN-VERIFY`
- **상태:** **RESOLVED** (2026-07-10)
- **연 시각:** 2026-07-10 (커밋 v0 e2e)
- **Revisit-trigger:** 성능 검증이 필요해지는 첫 순간, 또는 릴리스 태그 이전
- **Owner:** human

**상황:** `docs/00`의 v0 체크리스트 13개 중 12개를 통합 테스트로 검증했다. 남은 하나가 이것이다.

**발견:** 블로킹 격리를 **구조로는** 검증했다. 색인·검색·이력 저장·감사 저장이 모두 `boundedElastic-` 스레드에서 실행되는 것을 단언했고, BlockHound가 논블로킹 스레드의 블로킹 I/O를 실제로 잡는다는 것도 확인했다. 그러나 **동시 요청을 쌓아 이벤트 루프가 굶지 않는지는 본 적이 없다.** "격리되어 있다"와 "부하에서 견딘다"는 다르다.

**판단 근거:** 충족한 CONTINUE 조건 — *S 항목이 걸리지 않은 성능 관심사이고, 검증에 부하 도구·런타임이 필요하다.* S13이 요구하는 것은 "블로킹 접근을 전용 계층에 격리"이고 그것은 코드와 테스트로 강제되어 있다. 부하 특성은 배포 환경(스레드풀 크기, DB 커넥션 풀, ES 응답 시간)에 좌우되므로 단위 테스트로 답할 수 있는 질문이 아니다.

> 해당 없음: 도달 가능한 보안 사안 · S 항목 · 데이터 정합성 · 결정 문서

**회의적인 리뷰어에게:** 리뷰어는 "체크리스트를 12/13으로 만들어놓고 완성이라 부르려는 것 아니냐"고 지적할 것이다. 답은 — 그래서 끄지 않았다는 것이다. 이 항목은 미완으로 남아 있고, 부하 테스트 없이는 끌 수 없다.

**제안하는 조치:** 동시 요청을 걸어 스트리밍이 직렬화되지 않는지 본다. BlockHound를 켠 채 이벤트 루프 블로킹을 확인한다.

**해결:** `LoadIsolationTest`로 스위트에 넣었다. 수동 측정이 아니라 회귀를 잡는 테스트다.

- 블로킹 검색을 200ms로 만들고 동시 요청 32개를 던진다. 직렬이면 최소 6400ms가 걸린다.
- 실제 **246ms** — 오버랩 배율 26배. 워밍업을 분리하기 전엔 2014ms였다(첫 측정의 88%가 JIT·Jackson 캐시·커넥션 풀·Hibernate 초기화).
- 부하 중 블로킹 호출의 실행 스레드가 전부 `boundedElastic-`임을 단언한다. 이벤트 루프 이름이 하나라도 섞이면 실패한다.
- BlockHound를 무장한 채 돌려, 이벤트 루프 블로킹이 있으면 요청 자체가 500으로 깨지게 했다.

**부하 테스트가 실제 버그를 잡았다.** `AppUserRepository.ensureExists`가 "찾고 없으면 삽입"이라, 같은 사용자의 첫 요청이 동시에 들어오면 전부 삽입해 unique 제약을 위반하고 500이 났다. `insert ... on conflict do nothing` 후 읽기로 고치고, 경합을 재현하는 테스트를 남겼다. 단일 요청 테스트로는 절대 드러나지 않는 버그였다.

## OQ-006 · 색인·재색인 행위를 감사 로그에 남길 것인가

- **태그:** `AUDIT` `SCOPE`
- **상태:** **ANSWERED** (2026-07-10, 사람이 결정)
- **연 시각:** 2026-07-10 (재색인 API 사이클)
- **Revisit-trigger:** 감사 요구가 실제로 제기되는 첫 순간(고객 보안 검토·규제 문의), 또는 v0 이후 첫 릴리스 계획
- **Owner:** human

**상황:** 재색인 API를 만들면서 `document.uploaded_by`가 항상 NULL임을 발견했다. 파보니 `AuditLogRepository`를 부르는 프로덕션 코드는 `ChatService` 하나뿐이고, `REQ-AUDIT-logging.md`에는 "색인"이라는 단어가 없다.

**발견:** 관리자가 `restricted` 문서를 색인하거나 재색인해도 **audit_log에 아무 흔적이 남지 않는다.** `S5`의 감사 로그는 질문·응답 중심으로 설계되어 있고(추적ID·요청자ID·질문·응답·근거), 색인 행위를 담을 필드가 없다. 색인은 관리자 행위이므로 회사별 설치형 배포에서 감사 대상이 될 여지가 있다.

**이 항목은 `AUDIT` 태그라 OPEN 상태로 존재할 수 없다**(`docs/08` §D.2 — 자동 차단 태그). 보류가 아니라 사람의 결정이 필요한 사안이므로, 에이전트가 묻고 사람이 답한 뒤 `ANSWERED`로 기록한다.

**사람의 결정 (2026-07-10):** **`uploaded_by`만 채운다.** 색인 행위를 `audit_log`에 남기지 않는다.

근거:
- `docs/03`이 이미 `uploaded_by FK→app_user`를 정의하고 있다. 그것을 채우는 일은 **새 스코프가 아니라 문서가 정한 것의 구현**이며 `REQ` 본문 수정이 필요 없다.
- `audit_log`에 색인을 남기려면 `REQ-AUDIT` 본문 수정(사람 승인)이 필요하고, 질문·응답 중심 스키마의 필드 의미가 달라진다. v0 스코프를 넘는다.
- 색인 행위는 서버 로그에 `trace_id`와 함께 남는다(REL-3). 다만 **로그는 보존 정책이 없으므로 장기 감사 근거가 되지 않는다.** 이 한계를 알고 받아들인다.

**구현:** 커밋 `ce47293` — `document.uploaded_by`에 업로더를 기록한다. 재업로드는 갱신하고, 재색인은 유지한다(`null`이 "기존 유지"). 커버 테스트: `PostgresDocumentRepositoryTest.업로더가_기록된다` · `재색인은_업로더를_유지한다` · `재업로드는_업로더를_갱신한다`, `IndexControllerTest.업로더가_기록되고_재색인이_바꾸지_않는다`. 기존 4개 문서는 NULL로 남는다(누가 올렸는지 알 수 없어 소급 채우지 않는다).

**남은 것:** `audit_log`에 색인을 남길지는 위 Revisit-trigger에서 다시 본다.

## OQ-007 · 감사 로그 조회의 접근 통제 자리가 코드에 없다

- **상태:** **ANSWERED** (2026-07-10, 사람이 결정)
- **연 시각:** 2026-07-10 (기획 문서 대비 전수 감사)
- **Revisit-trigger:** 감사 조회 요구가 실제로 제기되는 첫 순간(고객 보안 검토·규제 문의), 또는 감사 대시보드 착수

**상황:** `docs/04` SEC-5는 *"감사 로그 접근은 감사 권한자만(v0는 최소 구현, **접근 통제 자리 확보**)"* 를, `REQ-AUDIT` 기능 명세 4번은 *"감사 조회는 감사 권한자만(v0 최소 구현, 자리 확보)"* 를 요구한다.

**발견:** 코드에 **감사 로그를 읽는 경로가 아예 없다.** `AuditLogRepository`는 `void record(...)` 하나뿐이고 `AuditController`는 존재하지 않는다. 쓰기 전용이므로 노출 위험은 없지만, 문서가 말한 "자리 확보"(인가 매처·역할)도 없다.

**사람의 결정 (2026-07-10):** **지금은 두고 기록만 한다.** 읽기 경로가 없으므로 노출 위험이 없고, `REQ-AUDIT` 자신이 "v0 최소 구현"이라고 적었다. 감사 조회 API를 만들면 `REQ-AUDIT` 본문 수정(사람 승인)이 필요하고, `audit_log`에는 질문·응답 전문이 들어 있어 노출 표면이 커진다.

## OQ-008 · 프론트엔드에 토큰 갱신이 없다

- **태그:** `FRONTEND` `UX`
- **상태:** **ANSWERED** (2026-07-10, 사람이 결정 — 프론트 마감 때 함께)
- **연 시각:** 2026-07-10 (기획 문서 대비 전수 감사)
- **Revisit-trigger:** 프론트엔드 마감 착수
- **Owner:** human

**상황:** `docs/04` SEC-1이 *"Refresh는 Keycloak 위임"* 이라고 적었고 `S25`가 Keycloak OIDC를 결정했다.

**발견:** `frontend/auth.ts`는 로그인 직후 `account.access_token`만 세션에 담는다. `refresh_token`도, 만료 시각도, 갱신 콜백도 없다(grep 0건). Keycloak 기본 access token 수명이 5분이므로 **그 뒤 사용자는 401을 맞고 대화가 끊긴다.** 재로그인 외에 방법이 없다.

**보안 결함은 아니다.** 만료된 토큰은 거부되므로 fail-closed다. 가용성·사용성 문제다.

**판단 근거:** 충족한 CONTINUE 조건 — *프론트엔드 표면이고, 사람이 "프론트는 뼈대만 두고 마지막에 한번에 다듬겠다"고 정한 영역이다.* 백엔드는 만료 토큰을 정확히 401로 거부한다(`SecurityConfig`, 테스트로 강제).

> 해당 없음: 도달 가능한 보안 사안 · S 항목 · 데이터 정합성 · 결정 문서

**회의적인 리뷰어에게:** 리뷰어는 "SEC-1이 refresh를 명시했는데 미구현 아니냐"고 지적할 것이다. 답은 — SEC-1의 문장은 "refresh를 **Keycloak에 위임한다**"이고 백엔드는 위임했다(자체 세션·자체 갱신을 만들지 않았다). BFF가 그 위임을 사용하지 않을 뿐이며, 실패는 401로 닫힌다.

**사람의 결정 (2026-07-10):** 프론트엔드 마감 때 함께 한다. 개발 중에는 재로그인으로 버틴다.

**할 일:** `auth.ts`의 `jwt` 콜백에 `refresh_token`·`expires_at` 저장 → 만료 임박 시 Keycloak 토큰 엔드포인트로 갱신 → 실패 시 세션 무효화.

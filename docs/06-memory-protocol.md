# 06 · 메모리 프로토콜 (교훈 정리)

> TDD·구현 중 얻은 교훈을 체계적으로 남겨, 다음 세션과 다음 모듈이 같은 함정을 반복하지 않게 한다.
> 대상 파일: **`docs/LEARNINGS.md`** (append-only 로그).

## 세 개의 기록 — 어디에 쓰는가

기록 대상은 세 곳이고 경계는 한 줄이다.

> **아직 하거나 정해야 할 일이 남았으면 `docs/OPEN-QUESTIONS.md`, 남은 게 없는 확정된 사실이면 `docs/LEARNINGS.md`, 코드리뷰 결과는 `docs/reviews/`.**

- **`LEARNINGS.md`** — 확정된 지식. 과거형. 열린 고리가 없다. ("WebFlux는 `OncePerRequestFilter`가 아니라 `WebFilter`를 쓴다")
- **`OPEN-QUESTIONS.md`** — 미결 결정·작업. 상태와 다음 행동의 주인이 있다. 무중단 진행 중 보류한 것들이 여기 쌓인다(`docs/08`).
- **`docs/reviews/`** — 코드리뷰 결과. 시점의 스냅샷이다. 아래 §코드리뷰 기록 참조.

앞의 둘에 동시에 있지 않는다. OQ가 닫히면서 남는 지식이 있으면 그때 LEARNINGS로 옮긴다.

## 코드리뷰 기록

**코드리뷰를 돌리면 결과를 반드시 문서로 남긴다.** 리뷰는 한 번 읽고 흘려보내는 것이 아니라, 무엇이 언제 어떤 근거로 발견되었는지 남는 기록이어야 한다.

- **위치:** `docs/reviews/YYYY-MM-DD-<범위>.md` (예: `2026-07-10-full-codebase-review.md`)
- **머리에 적는다:** 리뷰 범위, 기준 커밋 sha, 방법(어떤 관점으로 어떻게 돌렸는가), 당시 테스트 통과 수.
- **모든 발견에 근거 등급을 붙인다.** 이것이 이 문서의 핵심 규칙이다.

| 등급 | 뜻 |
|---|---|
| `[실증]` | 재현했거나 실행해서 확인했다. 실험 절차와 관측 결과를 적는다. |
| `[코드]` | 코드를 읽어 확정했다. 실행 재현은 하지 않았다. `파일:줄`을 인용한다. |
| `[미검증]` | 리뷰어의 판단이다. 확인이 필요하다. |

**추측을 사실처럼 쓰지 않는다.** 그럴듯한 추론은 `[미검증]`이다. 등급을 붙일 수 없으면 발견으로 적지 않는다.

- 발견마다 **구체적 실패 시나리오**를 쓴다. "어떤 입력·상태에서 무엇이 잘못되는가."
- **이미 테스트로 강제되는 것은 발견이 아니다.** 그 테스트의 `파일:줄`을 인용해 그렇게 적는다.
- **테스트가 무엇을 강제하지 않는지도 발견이다.** "이 프로덕션 코드를 이렇게 바꿔도 통과한다"를 확인했다면 `[실증]`이다.
- 리뷰 문서는 **고치지 않는다.** 시점의 기록이다. 발견을 처리하면 커밋에서 그 리뷰 항목 번호(`R-1` 등)를 인용하고, 남는 지식은 `LEARNINGS.md`로, 남는 결정은 `OPEN-QUESTIONS.md`로 옮긴다.
- `docs/reviews/`는 읽기 전용 문서가 아니다. 에이전트가 새 리뷰 파일을 자유롭게 만든다(`docs/08 §F`).

## 왜 필요한가
Claude Code는 세션이 바뀌면 앞 세션의 시행착오를 잊는다. TDD 중 발견한 "이 라이브러리는 실제로 이렇게 동작한다", "이 경계는 이렇게 구현해야 한다", "이 함정이 있었다"를 기록하지 않으면 매번 다시 부딪힌다. 이 로그가 프로젝트의 축적된 경험이다.

## 언제 기록하나
다음이 발생하면 **그 자리에서** `docs/LEARNINGS.md`에 append:
1. 라이브러리·프레임워크의 실제 동작이 예상과 달랐을 때(Spring AI, ES nori, WebFlux, hwplib 등).
2. 경계(E)를 실제 코드로 구현하며 내린 구체적 방식·이유.
3. 테스트가 어려웠던 지점과 해결책(mock 전략, testcontainers 설정 등).
4. 정합성 제약(docs/02 C)을 코드로 강제한 방법.
5. 실패·삽질 후 알아낸 것(다음에 피할 함정).
6. 문서(docs/)와 실제 구현이 어긋나 문서를 고쳐야 했던 경우.

## 기록 형식
각 항목은 아래 형식으로. 간결하게, 재현 가능하게.

```markdown
## [YYYY-MM-DD] <모듈/주제> — <한 줄 요약>
- **상황:** 무엇을 하려다가
- **발견:** 무엇을 알게 됐나 (사실 위주, 추측이면 "추정"이라 명시)
- **결정/해결:** 그래서 어떻게 했나
- **영향:** 어떤 경계·결정·문서에 영향 (S/E/REQ 번호로)
- **다음 주의:** 앞으로 피할 것
```

예:
```markdown
## [2026-07-15] SEARCH — nori 하이브리드는 단일 쿼리로 가능
- **상황:** BM25(nori)와 벡터를 각각 호출해 앱에서 병합하려 함
- **발견:** ES 쿼리 하나에 BM25 절과 kNN 절을 함께 넣어 RRF로 결합 가능. 앱 레벨 병합 불필요.
- **결정/해결:** 단일 쿼리 + ES 내장 RRF 사용. ResultMerger는 그 위 얇은 어댑터로.
- **영향:** S11, E10, PERF-3
- **다음 주의:** 벡터 차원 불일치 시 조용히 실패 → 색인/검색 모델 일치 테스트 필수(S8-4)
```

## 규칙
- **사실과 추측을 구분한다.** 확인 안 된 건 "추정"이라 표기(뇌피셜 금지).
- 문서(docs/)의 결정을 바꿔야 할 교훈이면, LEARNINGS에 남기고 **관련 docs도 함께 갱신**(MAINT-3). 로그만 남기고 문서를 방치하지 않는다.
- 세션 시작 시 `docs/LEARNINGS.md`를 읽고 시작한다.
- 민감정보(키·토큰·개인정보)를 로그에 남기지 않는다.

## LEARNINGS.md 초기 상태
프로젝트 시작 시 빈 파일로 생성하고, 첫 헤더만 둔다:
```markdown
# LEARNINGS — TDD·구현 교훈 로그
> 형식은 docs/06-memory-protocol.md 참조. append-only.
```

## A. 판정 순서 (위에서부터, 처음 걸리는 것이 답)

surface된 항목마다 이 순서로 판정한다. 결과는 반드시 `HALT` / `SKIP` / `CONTINUE` 중 하나다.

1. 진행하려면 거버넌스 문서를 고쳐야 하는가? → **HALT**
2. Green을 만들려면 절대 규칙 7개(`CLAUDE.md` §3) 또는 S 항목(`docs/02`)을 어겨야 하는가? → **HALT**
3. 보안·인가·감사·임베딩모델 사안이면서 (현재 코드 경로로 도달 가능 **OR** 트레이드오프 필요 **OR** 기존 테스트를 완화해야 함)인가? → **HALT**
4. 사전 승인 목록(§E) 밖의 의존성을 추가하거나 compose 토폴로지(새 컨테이너·포트·서비스)를 바꾸는가? → **HALT**
5. 파괴적·비가역 작업(데이터 삭제, 스키마 드롭, 히스토리 재작성, force 조작)인가? → **HALT**
6. 통과하던 테스트가 red가 되었고, 테스트를 약화하지 않고는 복구할 수 없는가? → **HALT**
7. 기대 결과가 진짜 모호한가(서로 다른 계약을 함의하는 단언이 2개 이상 방어 가능)? → **SKIP**
8. 아직 만들지 않은 모듈에 의존하는가? → **SKIP** (미충족 의존성을 이름으로 기록)
9. 에이전트에게 없는 런타임으로만 검증 가능한가(`docker compose up` 등)? → **SKIP** (단위·슬라이스가 green이면 다른 시나리오로 계속)
10. 현재 시나리오의 Green에 필요한가? **예** → 지금 시나리오 안에서 한다. **아니오** → **CONTINUE-and-log**

`HALT`는 세션을 멈추고 사람에게 묻는다. `SKIP`은 그 시나리오만 보류하고 다음 시나리오로 간다. `CONTINUE`는 기록만 하고 그대로 나아간다.

### 동점 처리 (비대칭)

판정 1~6(상위 3개 밴드: 보안 / 데이터 정합성 / 결정 준수)에서 **HALT 조건이 거짓임을 증명하지 못하면 참으로 간주해 HALT**한다. 판정 7 아래(테스트 / 기능 / 속도)에서 모호하면 CONTINUE-and-log.

근거: `CLAUDE.md`의 충돌 순서는 곧 **틀렸을 때의 비용** 순위다. 잘못된 HALT는 사람의 몇 분과 질문 하나를 쓴다. 보안·정합성·결정에서 잘못된 CONTINUE는 **조용한 위반이 되어 회사별 설치형 배포에 실려 나간다.** 사람의 주의는 되돌리기 비싼 곳에, 에이전트의 자율은 틀려도 기록되고 싸게 되돌릴 수 있는 곳에 쓴다. 무중단은 하위 3개 밴드를 위한 것이지 상위 3개를 위한 것이 아니다.

## B. 문서가 사실과 다를 때

에이전트는 **사실 오류 정정이라도 결정 문서를 스스로 고치지 않는다.** 자기 "정정"이 실은 결정 변경인지 판단할 자격이 가장 없는 것이 에이전트 자신이고, 그 지점이 최대의 합리화 표면이다. 문서 수정은 사람이 승인하기 싸다.

판정식은 하나다.

> 문서의 **문자 그대로의 지시가 이 스택에서 실행 불가**하고, 같은 목적을 달성하는 메커니즘이 **정확히 하나**인가?

- **예 → 메커니즘 정정.** 코드는 의도를 달성하는 그 메커니즘으로 구현하고, LEARNINGS에 사실을 남기고, 문서 수정안을 OQ로 올린다. **코드는 계속 나아간다.**
  - 실제 사례: `docs/01`이 JWT 검증을 서블릿 전용 `OncePerRequestFilter`로 하라고 적었다. WebFlux에서 그 클래스는 실행 자체가 불가능하고, "필터 계층에서 권한을 구조로 강제한다"는 같은 목적을 이루는 리액티브 메커니즘은 `WebFilter` 하나뿐이다 → 메커니즘 정정.
- **아니오 → 결정 변경 → HALT.** 지시가 실행은 되는데 내가 다른 방식을 선호하거나, 목적을 이루는 방식이 여럿이고 각각 다른 계약을 함의하는 경우다.

정리하면, 결정의 **의도**를 올바른 메커니즘으로 만족시킬 수는 있다. 결정이 **무엇인지를 다시 정하는 일**은 절대 하지 않는다.

## C. 무중단 루프

### 1회 반복

1. **선행조건 확인** — 하나라도 실패하면 HALT
2. **시나리오 선택** — REQ 선언 순서상 다음 *미차단* 시나리오
3. **Red** — 그 시나리오의 테스트만 작성하고 실행해 **실패 이유가 "미구현"인지 확인**(오타·설정오류면 안 된다)
4. **Green** — 최소 구현. **전체** 스위트 실행. 전부 green
5. **Refactor** — 선택. 동작 보존. 스위트 green 유지
6. **기록** — 확정 사실은 `LEARNINGS.md`, 보류 항목은 `OPEN-QUESTIONS.md`, 시나리오 체크박스 `- [x]`
7. **커밋** — Conventional Commits, 한 모듈, 본문에 `Tests: N passed`와 이번에 연 OQ id
8. **한 줄 상태 보고 후 즉시 다음 반복.** *"계속할까요?"라고 묻지 않는다.*

### 선행조건 (전부 성립해야 다음 반복 시작)

- 작업 트리가 깨끗하다(직전 반복이 커밋됨)
- 직전 커밋이 실제로 실행되어 green이었다
- OPEN 상태 OQ가 10건 미만이다
- 자동 차단 태그(§D)를 단 OPEN OQ가 없다
- 다음 미차단 시나리오가 존재한다
- 그 시나리오의 Green 실패 카운터가 3 미만이다
- 이번 세션에 `LEARNINGS.md`와 OPEN OQ를 읽었다

### 강제 정지 조건과 그 숫자의 근거

| 조건 | 값 | 근거 |
|---|---|---|
| 한 시나리오의 **Green 시도 연속 실패** | **3** | 명백한 수정 1회, 대안 1회, 재검토 1회. 그 이상은 추측이다. (첫 Red는 정상 TDD 단계이므로 세지 않는다. **Green 시도 실패**만 센다.) |
| OPEN OQ **WIP 상한** | **10** (7에서 경고) | 1인 개발자가 한자리에서 정리할 수 있는 양. 넘으면 실제 리뷰 부채다. 로그가 무덤이 되는 것을 물리적으로 막는다. |
| **체크포인트** | 20회 반복 또는 한 모듈 큐 완료 중 먼저 | 실패가 아니라 컨텍스트 드리프트를 막는 숨고르기. 자연스러운 경계는 "모듈 완료"다. |

### 시나리오 선택과 모듈 의존

모듈 큐를 의존 순서(IDX → AUTH → SEARCH → CHAT → AUDIT)로 이어 붙이고 앞에서부터 집는다.

- **미충족 의존성이 이름으로 지목될 때만** 시나리오를 건너뛴다. 예: `REQ-IDX`의 "비-ADMIN 업로드 → 403"은 AUTH가 필요하다 → `BLOCKED`로 기록하고 다음 미차단 시나리오로 간다. AUTH가 도착하면 자동으로 풀린다.
- **어렵지만 할 수 있다는 이유로는 건너뛰지 않는다.** 모든 건너뜀은 사유와 함께 기록되어, 리뷰어가 회피 여부를 확인할 수 있어야 한다.

## D. 반(反)합리화 장치

가장 큰 위험은 에이전트가 "이건 안 막아도 돼"라고 스스로를 설득하는 것이다. 그것을 비싸고 검증 가능하게 만든다.

### 1. 반증 가능한 보류 선언

모든 OQ의 `판단 근거`에는 (a) 충족한 CONTINUE 조건의 이름과 (b) 다음 문장이 **문자 그대로** 들어가야 한다.

> `해당 없음: 도달 가능한 보안 사안 · S 항목 · 데이터 정합성 · 결정 문서`

정직하게 쓸 수 없으면 보류할 수 없다 → **HALT**. 분위기로는 grep 가능한 문장을 만족시킬 수 없다.

### 2. 자동 차단 태그

`SEC-REACHABLE` · `S-ITEM` · `DATA` · `AUTHZ` · `AUDIT` · `EMBED-MODEL` 태그가 붙은 항목은 **OPEN 상태로 존재할 수 없다.** 위험한 밴드에서는 판단 자체를 없앤다. 리뷰어가 grep 한 번으로 위반을 잡는다.

(도달 불가능한 보안 사안은 `SEC-WATCH` 태그를 쓴다. 반드시 도달 불가 근거를 호출자 grep으로 보이고, 도달 가능해지는 조건을 **트립와이어**로 적는다.)

### 3. 금지어 — 계속 진행의 근거로 쓸 수 없는 표현

"일단은", "임시로", "나중에 확인하면 됨", "괜찮을 것 같다", "낮은 위험"(도달 불가를 grep으로 보이지 않은 채), "아마도", "테스트는 나중에 추가", "이 정도면 충분". 코드의 맨 `TODO` 주석은 OQ 항목을 대신하지 않는다.

### 4. 금지 동작 — 전부 diff에서 grep 가능하다

`@Disabled` · `@Ignore` · `assumeTrue` · `xfail` · `.skip(` · 단언을 주석 처리 · red를 green으로 만들기 위해 단언을 느슨하게 완화 · 기댓값을 관측값으로 교체 · 예외를 삼켜 통과 · 테스트 안 조기 return · 테스트 삭제 · `|| true` · 헬스체크나 임계값 낮추기.

### 5. 획득한 Green

커밋 본문에 테스트 러너 요약(`Tests: N passed`)을 붙인다. 증거가 없으면 green이 아니고, green이 아니면 커밋하지 않는다.

### 6. 테스트는 추가만 가능하다

시나리오의 테스트가 커밋된 뒤에는 단언을 **추가**만 할 수 있다. 단언 수를 줄이는 diff는 사람이 답한 OQ를 인용하는 `test:` 커밋일 때만 허용된다.

### 7. "회의적인 리뷰어에게 설명하기"

보류하기 전에 한 문장을 쓴다. *"내가 대충 넘어가는 걸 잡아내려는 리뷰어라면 ___를 지적할 것이고, 내 답은 ___이다."* 답이 빈약하면 HALT.

### 8. 스코프를 스스로 부여하지 않는다

일을 정당화하려고 REQ 문서에 시나리오를 추가하지 않는다. **제안**은 OQ로만 한다. 새 스코프는 사람에게서 온다.

## F. 문서 편집 권한

| 파일 | 에이전트 권한 |
|---|---|
| `docs/LEARNINGS.md`, `docs/OPEN-QUESTIONS.md` | 자유롭게 쓴다 |
| `docs/requirements/REQ-*.md`, `docs/00-overview.md` | **`- [ ]` → `- [x]` 만.** 다른 문자 변경 금지 |
| `CLAUDE.md`, `docs/01`~`docs/08` | 읽기 전용 |

사람이 승인한 문서 변경은 `LLMHUB_DOC_EDIT_APPROVED=1 git commit ...`으로 훅을 통과시킨다. 이 탈출구는 **사람이 명시적으로 승인했을 때만** 쓴다.

규칙은 `.githooks/pre-commit`이 강제한다. 문서로만 있는 규칙은 합리화 앞에서 무력하다.
